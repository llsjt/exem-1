# 大文件分片上传与断点续传 MVP 执行方案

## 1. 技术栈选型

### 1.1 前端

- 框架：`Vue 3`
- 语言：`TypeScript`
- 构建工具：`Vite`
- HTTP 请求：`XMLHttpRequest`
- 文件 Hash：`spark-md5`
- 测试：`Vitest`

选择理由：

- `Vue 3 + Vite` 适合快速搭建上传交互页面。
- `TypeScript` 有利于约束上传任务状态、接口参数和分片信息。
- `XMLHttpRequest.upload.onprogress` 原生支持上传进度事件，是实现真实上传进度的明确选型；`fetch` 不直接支持可靠的上传进度，不作为 MVP 上传实现。
- `spark-md5` 支持前端分片增量计算 Hash，适合大文件秒传判断。

### 1.2 后端

- 运行时：`Node.js 20+`
- 框架：`Express`
- 语言：`TypeScript`
- 分片解析：`multer`
- 文件操作：`fs.createWriteStream` / `fs.promises`
- 参数校验：`zod`
- 定时任务：`node-cron`
- 测试：`Vitest` / `Supertest`

选择理由：

- `Node.js + Express` 与前端同为 TypeScript 技术栈，适合全栈 MVP 快速落地。
- `fs.createWriteStream` 能流式写入分片和合并文件，避免一次性读入内存。
- `multer` 负责接收 `multipart/form-data` 分片文件，建议使用 `diskStorage` 并将目标目录配置到 `storage/chunks/{fileHash}/` 所在磁盘，避免从系统临时目录跨分区移动导致复制开销。
- `node-cron` 负责轻量定时清理。
- MVP 不接数据库，后端只依赖文件系统即可完成分片暂存、状态查询和合并。

### 1.3 存储

- 分片状态存储：本地文件系统目录结构。
- 分片元信息：`meta.json`。
- 合并后文件：本地文件系统。
- 数据库：MVP 不接入。

## 2. 已明确决策

| 待确认项 | 决策 | 理由 |
| --- | --- | --- |
| 登录鉴权 | MVP 不做 | 题目未要求，避免偏离核心上传流程 |
| 多用户隔离 | MVP 不做 | 无登录鉴权时不引入用户维度 |
| 返回文件访问地址 | 做，返回相对路径 | 便于验证合并成功和下载结果 |
| 页面刷新后续传 | MVP 不做 | 需要持久化前端任务和重新绑定本地文件，复杂度较高 |
| 文件标识方式 | 使用文件内容 Hash | 天然支持秒传判断和分片目录定位 |
| 合并触发方式 | 前端单独发起合并请求 | 流程更可控，避免最后分片并发到达导致竞态 |
| 分片状态存储 | 文件系统目录结构 | MVP 不引入数据库，降低依赖 |
| MIME Type | 传递，但不强校验 | 可用于下载响应，不作为安全校验依据 |
| 用户 ID / 目录信息 | MVP 不传 | 无多用户和目录管理需求 |
| 接入数据库 | MVP 不接 | 文件系统足够支撑必做功能 |
| 网络中断恢复 | 做基础恢复，不做自动续传 | 单片失败最多重试 2 次，仍失败后进入 `failed`，用户点击继续时查询后端状态并补传缺失分片 |
| HTTP 上传实现 | 使用 `XMLHttpRequest` | 需要 `upload.onprogress` 获取真实上传进度，`fetch` 不作为 MVP 选型 |
| Hash 算法 | `SparkMD5` 增量计算完整文件 MD5 | 性能足够，支持秒传和目录定位 |
| 并发控制 | 自定义队列 + 信号量 | 比通用 Promise 池更方便接入暂停、继续、取消和补传 |
| 单片失败处理 | 自动重试 2 次，仍失败则任务进入 `failed` | 控制复杂度，同时避免偶发网络抖动直接失败 |
| 取消清理 | 前端主动调用 `DELETE /api/uploads/{fileHash}` | 立即删除临时分片，不依赖定时清理兜底 |
| 部署形态 | MVP 只支持单实例部署 | 本地文件系统与 `node-cron` 不适合多实例共享状态 |

## 3. 系统数据流

主流程：

```text
文件选择/拖拽
  -> Hash Worker 使用 SparkMD5 分片增量计算文件 Hash
  -> 前端展示 hashing 状态和 Hash 计算进度
  -> 调用秒传检查接口
  -> 如果文件已存在，直接返回文件访问路径
  -> 如果文件不存在，使用 File.prototype.slice 切片
  -> 上传调度器使用队列 + 信号量控制最多 3 个分片并发上传
  -> 每个分片通过 XMLHttpRequest 上传并监听 upload.onprogress
  -> 后端保存分片到 storage/chunks/{fileHash}/
  -> 前端统计真实上传进度，进度阶段从 hashing 切换为 uploading
  -> 所有分片上传完成后，前端发起 merge 请求
  -> 后端校验分片完整性
  -> 后端按 chunkIndex 顺序流式合并
  -> 后端返回相对文件访问路径
```

暂停与继续流程：

```text
点击暂停
  -> 前端取消正在上传的请求
  -> 上传任务状态变为 paused

点击继续
  -> 前端调用状态查询接口
  -> 后端返回已存在的分片序号
  -> 前端过滤已上传分片
  -> 只补传缺失分片
```

网络中断恢复流程：

```text
分片上传请求因网络错误失败
  -> 当前分片最多自动重试 2 次
  -> 仍失败时判定为网络/服务异常
  -> uploadScheduler 停止继续派发新分片
  -> 上传任务状态变为 failed
  -> 用户恢复网络后点击继续
  -> 前端调用状态查询接口
  -> 后端只返回已经完整落盘的正式 .part 分片
  -> 前端重置未确认分片的进度
  -> 只补传后端缺失的分片
```

取消流程：

```text
点击取消
  -> 前端调用活跃 XMLHttpRequest 实例的 xhr.abort()
  -> 前端调用取消接口
  -> 后端删除 storage/chunks/{fileHash}/
  -> 上传任务状态变为 canceled
```

进度阶段：

- `hashing`：Hash Worker 每完成一个分片读取后通过 `postMessage` 回传 `{ loadedChunks, totalChunks, percent }`。
- `uploading`：每个分片的 `XMLHttpRequest.upload.onprogress` 回传当前分片已上传字节数，由 `progressCalculator.ts` 汇总总进度。
- 继续上传时，前端以 `GET /api/uploads/{fileHash}/status` 返回的 `uploadedChunks` 为准，已确认分片按完整大小计入进度，未确认分片进度重置为 `0`。
- `merging`：前端发起 `POST /api/uploads/merge` 后进入合并中状态，展示“正在合并...”，等待同步 merge 请求返回后进入 `success` 或 `failed`。

## 4. 后端接口设计

### 4.1 秒传检查

```http
POST /api/uploads/check
Content-Type: application/json
```

请求体：

```json
{
  "fileHash": "string",
  "fileName": "video.mp4",
  "fileSize": 1073741824,
  "chunkSize": 5242880,
  "totalChunks": 205,
  "mimeType": "video/mp4"
}
```

响应体：

```json
{
  "exists": false,
  "uploadedChunks": [0, 1, 2],
  "fileUrl": null
}
```

规则：

- 如果 `storage/files/{fileHash}/file` 存在，返回 `exists=true` 和 `fileUrl`。
- 如果文件未合并但存在部分分片，返回 `uploadedChunks`。
- 如果完全不存在，返回空的 `uploadedChunks`。
- `fileHash` 必须符合后端允许的 Hash 格式，例如 MD5 场景下使用 `[a-fA-F0-9]{32}`，禁止直接使用未校验的路径参数拼接文件路径。
- `fileSize`、`chunkSize`、`totalChunks` 必须为正数，并且 `totalChunks` 应与 `Math.ceil(fileSize / chunkSize)` 一致。

### 4.2 查询上传状态

```http
GET /api/uploads/{fileHash}/status
```

响应体：

```json
{
  "fileHash": "string",
  "status": "UPLOADING",
  "uploadedChunks": [0, 1, 2]
}
```

状态：

- `UPLOADING`：存在临时分片目录。
- `MERGING`：分片正在合并中。
- `MERGED`：已存在合并后文件。
- `NOT_FOUND`：无分片、无合并文件。

规则：

- 状态查询只统计正式的 `{chunkIndex}.part` 文件，不统计 `{chunkIndex}.part.tmp`。
- 如果已存在合并文件，优先返回 `MERGED`。
- 如果临时目录存在但缺少 `meta.json`，返回错误或按异常临时目录处理，不直接判定为完整上传状态。
- 状态查询接口应更新 `meta.json.updatedAt` 或单独维护 `lastAccessedAt`，表示用户仍在恢复上传流程中，避免清理任务在用户刚点击继续但尚未上传新分片时误删临时目录。

### 4.3 上传单个分片

```http
POST /api/uploads/chunks
Content-Type: multipart/form-data
```

表单字段：

- `fileHash`
- `fileName`
- `fileSize`
- `chunkSize`
- `totalChunks`
- `chunkIndex`
- `mimeType`
- `chunk`

响应体：

```json
{
  "chunkIndex": 0,
  "uploaded": true
}
```

规则：

- 分片最终保存为 `storage/chunks/{fileHash}/{chunkIndex}.part`。
- 写入时先保存到 `storage/chunks/{fileHash}/{chunkIndex}.part.tmp`，写入完成并关闭文件流后，再原子重命名为 `{chunkIndex}.part`。
- Windows 环境下 `fs.promises.rename` 在目标文件已存在时可能抛出 `EEXIST`，实现时需要先确认正式 `.part` 是否已存在且大小合法；如需覆盖，应先删除旧文件或写入新的临时文件名后再替换，避免重复上传导致异常。
- 状态查询和断点续传只认可正式 `.part` 文件，避免半截分片被误判为已上传。
- 第一次上传时写入或更新 `storage/chunks/{fileHash}/meta.json`，更新 `meta.json` 时也使用临时文件加原子替换。
- 重复上传同一分片时保持幂等，确认正式 `.part` 已存在且大小合法后返回成功；如发现 `.tmp` 残留，可覆盖写入。
- `chunkIndex` 必须满足 `0 <= chunkIndex < totalChunks`。
- 非最后一个分片的实际大小应等于 `chunkSize`，最后一个分片允许小于或等于 `chunkSize`。
- `fileHash`、`fileName`、`fileSize`、`chunkSize`、`totalChunks` 必须与已有 `meta.json` 保持一致。

### 4.4 合并分片

```http
POST /api/uploads/merge
Content-Type: application/json
```

请求体：

```json
{
  "fileHash": "string",
  "fileName": "video.mp4",
  "fileSize": 1073741824,
  "totalChunks": 205,
  "mimeType": "video/mp4"
}
```

响应体：

```json
{
  "fileHash": "string",
  "fileName": "video.mp4",
  "fileSize": 1073741824,
  "status": "MERGED",
  "fileUrl": "/api/files/string"
}
```

规则：

- 合并前必须校验 `0.part` 到 `{totalChunks - 1}.part` 全部存在。
- 合并前必须校验请求参数与 `storage/chunks/{fileHash}/meta.json` 一致。
- `mergeService.ts` 需要对同一个 `fileHash` 加互斥锁，避免重复点击或并发请求导致重复合并。
- 合并开始时将临时元信息状态更新为 `MERGING`，清理任务不得删除 `MERGING` 状态目录。
- 合并时按 `chunkIndex` 从小到大流式写入。
- 合并结果先写入 `storage/files/{fileHash}/file.tmp`，写入完成后原子重命名为 `storage/files/{fileHash}/file`。
- 合并后的元信息保存到 `storage/files/{fileHash}/meta.json`。
- 合并成功后删除临时分片目录。
- 如果合并中途失败，例如磁盘空间不足或文件写入异常，删除未完成的 `file.tmp`，保留 `storage/chunks/{fileHash}/` 以便修复后重新发起合并。
- 合并接口需要幂等：如果正式文件已存在，直接返回 `MERGED` 和 `fileUrl`。

### 4.5 取消上传

```http
DELETE /api/uploads/{fileHash}
```

响应体：

```json
{
  "fileHash": "string",
  "canceled": true
}
```

规则：

- 删除 `storage/chunks/{fileHash}/`，立即释放已上传但未合并的临时分片。
- 不删除 `storage/files/{fileHash}/`。
- 如果文件已经合并完成，取消接口只清理临时目录，不删除正式文件。

### 4.6 访问文件

```http
GET /api/files/{fileHash}
```

规则：

- 读取 `storage/files/{fileHash}/file`。
- 使用 `meta.json` 中的 `fileName` 设置下载文件名。
- 使用 `meta.json` 中的 `mimeType` 设置 `Content-Type`，但不强校验。
- `fileName` 写入响应头前需要做基础转义，避免响应头注入。
- 文件不存在、`meta.json` 缺失或文件不可读时返回统一错误响应。

### 4.7 统一错误响应

所有接口错误响应使用统一结构：

```json
{
  "code": "MISSING_CHUNK",
  "message": "缺少分片 12",
  "details": {
    "fileHash": "string",
    "chunkIndex": 12
  }
}
```

常见错误码：

- `INVALID_ARGUMENT`：参数缺失、格式错误或数值范围非法。
- `INVALID_FILE_HASH`：`fileHash` 格式非法。
- `UPLOAD_NOT_FOUND`：上传任务或临时目录不存在。
- `MISSING_CHUNK`：合并时缺少必要分片。
- `META_CONFLICT`：请求参数与 `meta.json` 不一致。
- `MERGE_IN_PROGRESS`：同一文件正在合并。
- `FILE_NOT_FOUND`：访问合并文件时文件不存在。
- `STORAGE_ERROR`：文件系统读写失败。

### 4.8 核心接口契约汇总

MVP 前后端至少围绕以下 4 类核心接口并行开发：

| 能力 | 接口 | 关键入参 | 关键出参 |
| --- | --- | --- | --- |
| 秒传与续传检查 | `POST /api/uploads/check` | `fileHash`、`fileName`、`fileSize`、`chunkSize`、`totalChunks` | `exists`、`uploadedChunks`、`fileUrl` |
| 分片上传 | `POST /api/uploads/chunks` | `fileHash`、`chunkIndex`、`chunk` | `chunkIndex`、`uploaded` |
| 状态查询 | `GET /api/uploads/{fileHash}/status` | `fileHash` | `status`、`uploadedChunks` |
| 合并/取消 | `POST /api/uploads/merge` / `DELETE /api/uploads/{fileHash}` | `fileHash`、`totalChunks` | `MERGED`、`fileUrl` 或 `canceled` |

## 5. 文件系统结构

```text
storage/
  chunks/
    {fileHash}/
      meta.json
      0.part.tmp
      0.part
      1.part
      2.part
  files/
    {fileHash}/
      meta.json
      file.tmp
      file
```

临时分片元信息示例：

```json
{
  "fileHash": "string",
  "fileName": "video.mp4",
  "fileSize": 1073741824,
  "chunkSize": 5242880,
  "totalChunks": 205,
  "mimeType": "video/mp4",
  "schemaVersion": 1,
  "status": "UPLOADING",
  "createdAt": "2026-05-26T14:00:00",
  "updatedAt": "2026-05-26T14:10:00",
  "lastAccessedAt": "2026-05-26T14:10:30"
}
```

合并后文件元信息示例：

```json
{
  "fileHash": "string",
  "fileName": "video.mp4",
  "fileSize": 1073741824,
  "mimeType": "video/mp4",
  "schemaVersion": 1,
  "status": "MERGED",
  "fileUrl": "/api/files/string",
  "completedAt": "2026-05-26T14:20:00"
}
```

存储一致性规则：

- `fileHash` 只能作为 MVP 文件内容标识，不作为长期业务主键；`meta.json` 预留 `schemaVersion`，后续可扩展 `uploadId`、`ownerId`、`bizType` 等字段。
- `storageService` 拼接路径前必须校验 `fileHash`，并确保最终路径仍在配置的上传根目录下。
- 同一个 `fileHash` 的分片写入、`meta.json` 更新、合并和清理需要通过进程内 `Map<string, Mutex>` 或文件锁互斥。
- 清理任务不得删除已存在 `storage/files/{fileHash}/file` 的文件目录。
- 清理任务只清理 `storage/chunks/` 下超过过期时间、非 `MERGING`、当前无上传/合并锁的临时目录。
- 对于同一文件重复上传，MVP 语义为：只要 `storage/files/{fileHash}/file` 已存在，直接返回秒传成功，不再复制一份文件。

## 6. 模块拆分

### 6.1 前端模块

```text
frontend/
  src/
    components/
      UploadDropzone.vue
      UploadTaskList.vue
      UploadTaskItem.vue
    workers/
      hash.worker.ts
    services/
      uploadApi.ts
      xhrUploadClient.ts
    stores/
      uploadStore.ts
    utils/
      chunkFile.ts
      uploadScheduler.ts
      progressCalculator.ts
```

职责：

- `UploadDropzone.vue`：文件选择与拖拽入口。
- `UploadTaskList.vue`：上传任务列表容器。
- `UploadTaskItem.vue`：单个任务状态、进度条和操作按钮。
- `hash.worker.ts`：在 Worker 内读取 `File` 分片，优先使用 `file.slice().arrayBuffer()`；也可使用 Worker 专属的 `FileReaderSync`。不建议把整个文件一次性读入内存。通过 `spark-md5` 增量计算文件 Hash，并用 `postMessage` 回传 Hash 进度。
- `uploadApi.ts`：封装后端上传接口。
- `xhrUploadClient.ts`：基于 `XMLHttpRequest` 封装分片上传、`upload.onprogress`、`xhr.abort()` 取消和单片重试。
- `uploadStore.ts`：管理当前上传任务状态。
- `chunkFile.ts`：封装 `File.prototype.slice` 切片逻辑。
- `uploadScheduler.ts`：基于队列 + 信号量控制并发上传，负责暂停、继续、取消、失败恢复和缺失分片补传。
- `progressCalculator.ts`：根据 Hash 阶段进度和各分片上传字节数计算阶段进度。

### 6.2 后端模块

```text
backend/
  src/
    app.ts
    routes/
      uploadRoutes.ts
      fileRoutes.ts
    controllers/
      uploadController.ts
      fileController.ts
    services/
      uploadService.ts
      mergeService.ts
      cleanupService.ts
    storage/
      storageService.ts
      lockManager.ts
    config/
      uploadConfig.ts
      cronConfig.ts
    middlewares/
      uploadMiddleware.ts
      errorHandler.ts
    types/
      uploadTypes.ts
      errorTypes.ts
```

职责：

- `uploadRoutes.ts`：注册 `/api/uploads/*` 上传接口。
- `fileRoutes.ts`：注册 `/api/files/{fileHash}` 文件访问接口。
- `uploadController.ts`：解析 HTTP 请求并调用上传服务。
- `fileController.ts`：读取合并后文件并设置下载响应头。
- `uploadService.ts`：秒传检查、状态查询、分片接收、取消上传。
- `mergeService.ts`：分片完整性校验、互斥锁控制和流式合并。
- `cleanupService.ts`：孤儿分片清理。
- `storageService.ts`：文件系统读写、目录扫描、元信息读写、临时文件原子替换。
- `lockManager.ts`：维护同一 `fileHash` 下上传、合并、清理的互斥。
- `uploadMiddleware.ts`：基于 `multer` 接收单个分片文件。
- `errorHandler.ts`：统一处理参数错误、业务异常和文件系统异常。
- `uploadTypes.ts` / `errorTypes.ts`：定义 Request、Response、状态和错误结构。

## 7. 执行步骤与阶段测试

开发顺序必须围绕 `fileHash` 前置展开。`fileHash` 是分片目录、秒传检查、断点续传、合并请求的共同主键，因此 Hash 计算不能放到分片上传之后。

以下阶段按依赖关系编排，不要求所有工作单线程串行。能并行的部分拆给多个子 agent 同时推进，但每个子 agent 的写入范围必须独立，避免互相覆盖。

### 7.1 必须串行的主干关卡

| 关卡 | 必须完成内容 | 阻塞原因 | 完成后解锁 |
| --- | --- | --- | --- |

### 7.4 并行执行约束

- 每个子 agent 必须有明确写入范围，前端、后端、测试文件尽量分离。
- `uploadTypes.ts`、`errorTypes.ts`、接口路径、状态枚举由架构/契约 agent 先定，后续子 agent 不得擅自改契约。
- 前端调度 agent 和暂停恢复 agent 共享 `uploadScheduler.ts` 时，需要先约定状态机接口；若无法拆清，暂停恢复应等待调度基础完成。
- 后端合并 agent 和清理 agent 共享 `lockManager.ts` 与 `meta.status`，必须先约定 `UPLOADING`、`MERGING`、`MERGED`、`NOT_FOUND` 的状态含义。
- 并行子 agent 完成后必须先跑各自模块测试，再进入下一串行关卡。
- 串行关卡只做集成确认，不重新发明模块实现。

### 阶段 1：确定接口契约与项目骨架

执行内容：

- 创建 `frontend/` Vite Vue 3 项目。
- 创建 `backend/` Node.js Express TypeScript 项目。
- 配置前端代理 `/api` 到后端 `8080`。
- 固化核心接口：`check`、`status`、`chunks`、`merge`、`delete`、`files`。
- 配置后端上传根目录 `storage/`。
- 在根目录或后端目录配置 `.gitignore`，忽略 `storage/`、`storage/chunks/`、`storage/files/`、测试生成的大文件和临时分片。

阶段测试：

- 前端执行 `rtk npm run build`，确认构建通过。
- 后端执行 `rtk npm test`，确认 Express 应用和基础路由可加载。
- 使用 `Supertest` 访问健康检查或基础接口，确认服务可运行。
- 确认 `storage/` 下测试产生的临时文件不会进入版本控制。

### 阶段 2：实现前端 Hash Worker 与文件切片

执行内容：

- 实现 `hash.worker.ts`。
- 在 Worker 内使用 `file.slice().arrayBuffer()` 分片读取 `File`，或使用 Worker 专属 `FileReaderSync`；不使用一次性全量读取。
- 使用 `spark-md5` 增量计算完整文件 MD5。
- Worker 通过 `postMessage` 回传 Hash 计算进度。
- 实现 `chunkFile.ts`，使用 `File.prototype.slice` 生成分片描述。
- 默认分片大小 `5MB`，最后一个分片允许小于 `5MB`。

阶段测试：

- 同一文件多次计算 Hash 一致。
- 切片数量等于 `Math.ceil(file.size / chunkSize)`。
- 最后一个分片大小正确。
- 覆盖空文件、小于 `5MB`、刚好 `5MB`、`5MB + 1 byte`、接近 `1GB` 的切片用例。
- Hash 计算期间任务状态为 `hashing`，界面能显示“正在计算文件指纹”和计算进度。
- Hash 计算过程中内存占用不随文件大小线性增长。

### 阶段 3：实现后端文件系统存储层

执行内容：

- 实现上传根目录初始化。
- 实现目录规则：临时分片目录 `storage/chunks/{fileHash}/`，合并文件目录 `storage/files/{fileHash}/`。
- 实现分片文件命名规则：`{chunkIndex}.part.tmp` 和 `{chunkIndex}.part`。
- 实现 `meta.json` 写入、读取、更新。
- 实现已上传分片扫描，状态来源为文件系统目录，不依赖数据库。
- 实现 Windows 下重复上传同一分片时的 rename 兼容处理，避免 `EEXIST` 导致接口失败。

阶段测试：

- 能创建 `storage/chunks/{fileHash}/`。
- 能写入并读取 `meta.json`。
- 能扫描已有 `.part` 文件并返回分片序号。
- 写入分片时先生成 `.part.tmp`，完成后再生成正式 `.part`。
- 残留 `.part.tmp` 不会被扫描为已上传分片。
- Windows 下目标 `.part` 已存在时，重复上传仍能保持幂等并返回成功。
- 非法 `fileHash` 不能逃逸上传根目录。

### 阶段 4：实现秒传检查与状态查询接口

执行内容：

- 实现 `POST /api/uploads/check`。
- 实现 `GET /api/uploads/{fileHash}/status`。
- 支持 `NOT_FOUND`、`UPLOADING`、`MERGING`、`MERGED` 四种状态。

阶段测试：

- 文件不存在时返回 `exists=false` 和空分片列表。
- 存在部分分片时返回已上传分片列表。
- 已合并文件存在时返回 `exists=true` 和 `fileUrl`。
- `fileHash` 缺失或格式非法时返回 `INVALID_FILE_HASH`。
- `fileSize <= 0`、`chunkSize <= 0`、`totalChunks <= 0` 时返回 `INVALID_ARGUMENT`。
- 查询未合并任务状态时更新 `updatedAt` 或 `lastAccessedAt`，防止恢复流程中被清理任务误删。

### 阶段 5：实现前端上传 API 与 XHR 客户端

执行内容：

- 实现 `uploadApi.ts`，封装 `check`、`status`、`merge`、`delete`。
- 实现 `xhrUploadClient.ts`，使用 `XMLHttpRequest` 上传单个分片。
- 使用 `XMLHttpRequest.upload.onprogress` 获取分片实时上传字节数。
- 每个分片上传创建一个 `XMLHttpRequest` 实例。
- 使用 `Map<chunkIndex, XMLHttpRequest>` 管理活跃请求。
- 暂停或取消时调用对应实例的 `xhr.abort()`，而不是使用 `AbortController`。
- 单片失败自动重试 `2` 次，仍失败则抛出错误。

阶段测试：

- 秒传检查请求参数正确。
- 分片上传请求参数正确。
- `upload.onprogress` 能回传分片上传字节数。
- 主动 `xhr.abort()` 后请求被中止，并能区分用户暂停和网络错误。
- 活跃请求从 `Map<chunkIndex, XMLHttpRequest>` 中正确移除，避免内存泄漏和重复取消。
- 单片失败重试次数不超过 `2` 次。

### 阶段 6：实现后端分片接收与暂存

执行内容：

- 实现 `POST /api/uploads/chunks`。
- 使用 `multer.diskStorage` 接收字段名为 `chunk` 的分片文件，并尽量直接落到 `storage/chunks/{fileHash}/` 所在磁盘。
- 校验 `chunkIndex` 范围。
- 将分片先写入 `{chunkIndex}.part.tmp`，写入完成后原子重命名为 `{chunkIndex}.part`。
- 更新 `meta.json` 的 `updatedAt`。

阶段测试：

- 上传单个分片后文件落盘。
- 重复上传同一分片仍返回成功。
- 非法 `chunkIndex` 返回参数错误。
- 缺少 `chunk` 文件时返回 `INVALID_ARGUMENT`。
- 非最后一个分片大小不等于 `chunkSize` 时返回参数错误；`multer` 的 `req.file.size` 是解析后的纯文件内容大小，不包含 multipart 编码开销。
- `fileSize`、`chunkSize`、`totalChunks` 与已有 `meta.json` 不一致时返回 `META_CONFLICT`。
- 分片上传被中断时，不产生可被状态查询识别的正式 `.part` 文件。

### 阶段 7：前后端分片上传联调

执行内容：

- 前端完成 Hash 计算后调用 `POST /api/uploads/check`。
- 文件不存在时执行切片并上传少量分片。
- 后端落盘分片到 `storage/chunks/{fileHash}/`。
- 前端使用 `GET /api/uploads/{fileHash}/status` 验证已上传分片。

阶段测试：

- `fileHash` 能贯穿前端 Hash、秒传检查、分片目录和状态查询。
- 上传前 3 个分片后，状态查询返回 `[0, 1, 2]`。
- 上传同一分片多次不会破坏 `meta.json`。

### 阶段 8：实现并发调度与真实进度

执行内容：

- 实现 `uploadScheduler.ts`。
- 使用自定义队列 + 信号量控制最大并发数 `3`。
- 实现 `progressCalculator.ts`，区分 `hashing` 进度和 `uploading` 进度。
- 多个分片并发上传时，按分片已上传字节数汇总总上传进度。
- 上传进度公式为 `uploadProgress = sum(chunkUploadedBytes) / totalFileSize`，不能使用 `completedChunks / totalChunks`。

阶段测试：

- 同时上传请求数不超过 `3`。
- 总进度能根据各分片已上传字节数正确计算，并能平滑变化。
- 某分片重试时，该分片未确认的进度可被重置。
- Hash 阶段和上传阶段的 UI 状态不混淆。

### 阶段 9：实现暂停、继续、取消

执行内容：

- 暂停：中止正在上传的 XHR 请求，停止继续派发新分片，任务状态变为 `paused`。
- 继续：先调用 `GET /api/uploads/{fileHash}/status`，再只上传缺失分片。
- 取消：中止正在上传的 XHR 请求，并调用 `DELETE /api/uploads/{fileHash}`。
- 后端取消时立即删除 `storage/chunks/{fileHash}/`。

阶段测试：

- 暂停后正在上传的请求被取消。
- 暂停后不会继续派发新分片。
- 点击继续后先查询后端状态，再上传缺失分片。
- 点击取消后任务状态变为 `canceled`，后端临时分片目录被删除。
- 连续点击暂停、继续不会重复派发同一分片。

### 阶段 10：实现网络中断恢复与缺失分片补传

执行内容：

- 网络错误和用户主动暂停需要区分。
- 网络错误时任务进入 `failed`，不自动无限重试。
- `failed` 状态保留继续入口。
- 继续时复用状态查询和缺失分片补传流程。

阶段测试：

- 网络错误导致分片上传失败时，单片最多重试 `2` 次。
- 超过重试次数后任务状态变为 `failed`。
- `failed` 状态点击继续时，先调用 `GET /api/uploads/{fileHash}/status`，再补传缺失分片。
- 继续上传时以 `uploadedChunks` 为准，重置未被后端确认分片的进度。

### 阶段 11：实现合并与文件访问

执行内容：

- 实现 `POST /api/uploads/merge`。
- 合并前校验全部分片存在。
- 对同一 `fileHash` 加互斥锁。
- 按序流式合并为 `storage/files/{fileHash}/file.tmp`，完成后重命名为 `file`。
- 写入合并后的 `meta.json`。
- 实现 `GET /api/files/{fileHash}`。
- 前端发起 merge 请求后进入 `merging` 状态，显示“正在合并...”，等待同步响应返回。

阶段测试：

- 完整分片可以成功合并。
- 缺少任意分片时合并失败。
- 乱序上传的分片仍按 `chunkIndex` 顺序合并。
- 并发重复调用 `POST /api/uploads/merge` 时，只有一个请求执行合并，其余返回 `MERGE_IN_PROGRESS` 或已合并结果。
- 合并期间写入 `file.tmp`，完成后才生成正式 `file`。
- 合并中途失败时删除 `file.tmp`，保留临时分片目录。
- 合并期间内存不随文件大小线性增长，不能一次性读入所有分片。
- merge 请求期间前端状态为 `merging`，成功后变为 `success`，失败后变为 `failed`。

### 阶段 12：实现孤儿分片清理

执行内容：

- 实现 `cleanupService.ts`。
- 使用 `node-cron` 按配置频率执行清理任务。
- 清理超过配置时间仍未合并的 `storage/chunks/{fileHash}/`。
- 清理时避开 `MERGING` 状态和当前持有上传/合并锁的目录。
- 清理判断同时参考 `updatedAt` 和 `lastAccessedAt`，避免用户刚开始恢复上传但尚未产生新分片时被误删。

阶段测试：

- 清理任务能删除过期临时目录。
- 清理任务不会删除 `storage/files/` 下的已合并文件。
- 未过期的临时目录不会被清理。
- 仅调用状态查询但尚未上传新分片的活跃任务不会被清理。
- `MERGING` 状态或持有上传/合并锁的目录不会被清理。
- `meta.json` 缺失但存在 `.part` 文件的异常目录需要按清理策略处理，并记录错误日志。

### 阶段 13：MVP 联调验收与异常专项测试

执行内容：

- 完成完整上传链路联调。
- 验证秒传、暂停、继续、取消、网络中断恢复、合并、下载、清理。
- 补充后端接口异常矩阵测试。
- 补充同一 `fileHash` 下分片上传、合并、取消、清理之间的竞态测试。

阶段测试：

- 大文件可以完整上传成功。
- Hash 计算阶段显示指纹计算进度。
- 上传过程中进度条真实变化。
- 暂停后不会继续上传新分片。
- 继续后只补传缺失分片。
- 重复上传同一文件触发秒传。
- 合并成功后返回相对文件访问路径。
- 文件访问接口能下载合并后的文件。
- 取消上传后临时分片被删除。
- 超时孤儿分片能被清理。
- 上传过程中断网或模拟分片请求失败后，恢复网络并点击继续，只上传后端缺失分片。
- 使用接近 `1GB` 的测试文件或等价大文件模拟数据，验证分片数量、上传并发、合并结果和内存占用。
- 合并后文件大小和 Hash 与原始文件一致。
- 前端执行 `rtk npx vitest run`，后端执行 `rtk npm test`，全部通过后再进入最终验收。

- 默认孤儿分片过期时间：`24h`
- 默认清理任务频率：每 `30min`
- 单片失败最大重试次数：`2`
- 前端配置建议：
  - `VITE_UPLOAD_CHUNK_SIZE=5242880`
  - `VITE_UPLOAD_CONCURRENCY=3`
  - `VITE_UPLOAD_CHUNK_RETRY=2`
- 后端配置建议：
  - `UPLOAD_STORAGE_ROOT=storage`
  - `UPLOAD_CHUNK_SIZE=5242880`
  - `UPLOAD_CHUNK_EXPIRE_HOURS=24`
  - `UPLOAD_CLEANUP_CRON=*/30 * * * *`
  - `UPLOAD_MAX_CHUNK_SIZE=10485760`
  - `PORT=8080`

## 9. 业务场景处理规则

### 9.1 1GB 视频上传

- `1GB` 文件按 `5MB` 分片约 `205` 个分片。
- Hash 阶段进入 `hashing` 状态，显示“正在计算文件指纹”和 Hash 进度。
- 上传阶段进入 `uploading` 状态，最大并发数固定为 `3`。
- 上传进度只表示网络上传进度，不与 Hash 进度混算。

### 9.2 网络中断后恢复

- 网络错误和用户主动暂停必须区分。
- 用户主动暂停进入 `paused`，网络错误超过单片重试次数后进入 `failed`。
- MVP 不做自动恢复，用户点击继续后恢复上传。
- 继续时必须查询 `GET /api/uploads/{fileHash}/status`，并只补传后端缺失分片。

### 9.3 秒传

- MVP 不做多用户隔离。
- 如果 `storage/files/{fileHash}/file` 已存在，后端直接返回 `exists=true` 和 `fileUrl`。
- 前端提示秒传成功，不再重复上传和复制文件。

### 9.4 取消上传

- 前端取消时立即中止正在上传的 XHR 请求。
- 前端随后调用 `DELETE /api/uploads/{fileHash}`。
- 后端立即删除 `storage/chunks/{fileHash}/`。
- 取消逻辑不依赖定时清理任务。

### 9.5 合并异常

- 合并接口必须幂等。
- 合并过程加锁，避免重复合并。
- 合并失败时删除 `file.tmp`，保留临时分片目录，允许后续重新合并。
- 磁盘不足、文件系统异常等错误返回统一 `STORAGE_ERROR`。

## 10. 非 MVP 范围

以下内容第一版不做：

- 登录鉴权。
- 多用户隔离。
- 页面刷新后续传。
- 多文件批量上传。
- 多文件并行队列。
- 上传速度展示。
- 剩余时间估算。
- 复杂失败重试策略。
- 文件预览。
- 文件分享。
- 文件夹管理。
- 对象存储直传。
- 分布式部署。
- 数据库存储上传状态。

## 11. 实施规则

- 不一次性完成所有模块。
- 按第 7 节的串并行编排执行：串行关卡负责收敛接口和联调，并行子 agent 负责互不冲突的模块实现。
- 每完成一个模块，先执行该模块测试。
- 当前模块测试不通过，不进入下一模块。
- 后端接口优先实现，前端基于后端接口联调。
- `fileHash` 是系统主键，Hash Worker 必须在分片上传、状态查询、合并之前完成。
- 上传进度必须基于 `XMLHttpRequest.upload.onprogress`，不使用 `fetch` 实现 MVP 上传进度。
- 并发调度使用自定义队列 + 信号量，便于暂停、继续、取消和缺失分片补传。
- 不引入需求未明确要求的加分项。
- 不接数据库。
- 不做页面刷新后的续传。
- 不做登录鉴权和多用户隔离。
