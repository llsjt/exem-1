# 大文件分片上传与断点续传需求分析

## 1. 题目目标

实现一个支持上传大文件的云盘上传组件，重点偏后端能力建设。

核心目标：

- 支持上传大文件，例如 `1GB` 视频。
- 前端将文件切分为多个小分片后上传，不能通过单个 Ajax 请求直接上传完整文件。
- 上传过程中展示真实百分比进度条。
- 支持网络中断、暂停后的恢复上传。
- 恢复上传时只上传后端缺失的分片。
- 后端负责接收分片、暂存分片、合并完整文件。
- 后端需要定时清理长时间未合并的孤儿分片，释放磁盘空间。

## 2. 目标用户

产品用户：

- 需要上传大文件的云盘用户。
- 上传视频、压缩包、文档等较大文件的用户。
- 在网络不稳定环境下仍需要可靠上传体验的用户。

开发与考察对象：

- 前端开发者：重点考察 `Blob`、`File`、`FileReader`、`Web Worker`、分片上传、并发控制等能力。
- 后端开发者：重点考察文件流处理、临时状态管理、分片合并、孤儿分片清理等能力。

待确认：

- 是否需要登录鉴权。
- 是否需要多用户文件隔离。
- 上传完成后是否需要返回文件访问地址。
- 是否需要支持跨浏览器刷新后的任务恢复。

## 3. 必须完成的核心功能

### 3.1 前端功能

- 文件选择。
- 文件拖拽上传。
- 上传任务列表。
- 每个上传任务展示百分比进度条。
- 每个上传任务支持暂停、继续、取消操作。
- 使用 `File.prototype.slice` 将文件切分为多个分片。
- 控制分片并发上传数量，例如最多同时上传 `3` 个分片。
- 上传前计算整个文件的 Hash。
- Hash 计算建议放到 `Web Worker` 中，避免阻塞主线程。
- 上传前携带文件 Hash 请求后端，判断文件是否已存在。
- 如果文件已存在，直接提示“秒传成功”。
- 点击暂停时，中止正在进行的分片上传请求。
- 点击继续时，先向后端查询当前已收到哪些分片。
- 继续上传时只上传缺失分片。

### 3.2 后端功能

- 提供文件秒传检查能力。
- 提供已上传分片查询能力。
- 提供单个分片接收能力。
- 按文件标识暂存分片。
- 保存或识别分片序号。
- 在所有分片上传完成后，支持合并分片。
- 合并时按分片序号顺序流式写入，生成完整文件。
- 提供定时任务，清理长时间未合并的孤儿分片。

待确认：

- 文件唯一标识使用完整文件 Hash，还是额外生成上传任务 ID。
- 合并由最后一个分片触发，还是由前端单独发起合并请求。
- 分片临时状态存储在文件系统、数据库还是内存中。

## 4. 输入、输出、约束条件

### 4.1 输入

前端输入：

- 用户选择或拖拽的文件。
- 上传任务操作：暂停、继续、取消。

上传相关数据：

- 文件名。
- 文件大小。
- 文件 Hash。
- 分片大小。
- 分片序号。
- 总分片数。
- 单个分片二进制内容。

后端接口输入：

- 文件 Hash。
- 文件名。
- 文件大小。
- 分片序号。
- 总分片数。
- 分片内容。

待确认：

- 是否需要传 MIME Type。
- 是否需要传用户 ID。
- 是否需要传文件目录或业务归属信息。

### 4.2 输出

前端输出：

- 上传任务列表。
- 每个任务的上传状态。
- 每个任务的真实上传进度百分比。
- 秒传成功提示。
- 上传成功或失败提示。

后端输出：

- 文件是否已存在。
- 已上传分片序号列表。
- 单个分片上传结果。
- 文件合并结果。
- 错误信息。

待确认：

- 上传完成后是否需要返回文件下载地址。
- 合并成功后是否需要返回文件元数据。

### 4.3 约束条件

- 大文件不能通过单个 Ajax 请求整体上传。
- 必须使用分片上传。
- 必须控制分片上传并发数。
- 必须支持暂停和恢复上传。
- 恢复上传时必须查询后端已上传分片状态。
- 进度条需要基于真实上传进度计算。
- 后端合并文件时不能一次性把所有分片读入内存。
- 后端需要清理长时间未完成上传的临时分片。

## 5. MVP 具体范围

MVP 第一版只完成题目明确要求的必做能力。

### 5.1 MVP 必须做

- 单文件上传。
- 文件选择上传。
- 文件拖拽上传。
- 文件 Hash 计算。
- 上传前秒传检查。
- 固定分片大小，默认参考 `5MB`，具体值可配置。
- 固定分片并发数，默认参考 `3`，具体值可配置。
- 分片上传。
- 上传任务列表。
- 上传百分比进度展示。
- 暂停上传。
- 继续上传。
- 取消上传。
- 继续上传前查询后端已上传分片。
- 只补传缺失分片。
- 后端接收并暂存分片。
- 后端查询已上传分片。
- 后端按序合并分片。
- 后端定时清理长时间未合并的临时分片。

### 5.2 MVP 不做

- 多文件批量上传。
- 多文件同时上传队列。
- 上传速度展示。
- 剩余时间估算。
- 复杂失败重试策略。
- 文件预览。
- 文件分享。
- 文件夹管理。
- 多端同步。
- 登录鉴权。
- 多用户权限隔离。
- 对象存储直传。
- 分布式部署。
- 分片级别复杂校验。
- 上传完成后的文件管理后台。

待确认：

- MVP 是否必须支持页面刷新后的续传。
- MVP 是否必须持久化上传任务状态。
- MVP 是否需要真实接入数据库。

## 7. 潜在风险点

- Hash 计算性能风险：大文件 Hash 计算耗时较长，如果在主线程执行，可能导致页面卡顿。
- 进度统计风险：多个分片并发上传时，需要正确累计每个分片已上传字节数，否则进度条不准确。
- 暂停恢复一致性风险：前端认为某分片已上传，但后端尚未完成落盘，恢复时可能漏传。
- 分片合并顺序风险：后端必须按分片序号合并，不能按分片到达顺序合并。
- 临时文件清理风险：定时任务不能误删正在上传或即将合并的分片。

## 8. 建议项目结构

技术栈暂不限定，以下为职责拆分建议。

```text
project/
  frontend/
    components/
      UploadDropzone
      UploadTaskList
      UploadTaskItem
    workers/
      hash.worker
    services/
      uploadApi
    utils/
      chunkFile
      uploadScheduler
      progressCalculator

  backend/
    controllers/
      uploadController
    services/
      uploadService
      mergeService
      cleanupService
    storage/
      tempChunks/
      mergedFiles/
    jobs/
      cleanupChunksJob
    models/
      uploadTask
      chunkInfo
```

模块职责：

- `UploadDropzone`：负责文件选择与拖拽交互。
- `UploadTaskList`：负责展示上传任务列表。
- `UploadTaskItem`：负责单个任务的状态、进度和操作按钮。
- `hash.worker`：负责计算文件 Hash。
- `uploadApi`：负责封装秒传检查、分片上传、查询分片、合并请求。
- `chunkFile`：负责文件切片。
- `uploadScheduler`：负责分片并发调度。
- `progressCalculator`：负责计算真实上传进度。
- `uploadController`：负责接收上传相关 HTTP 请求。
- `uploadService`：负责分片状态判断与上传流程编排。
- `mergeService`：负责按序流式合并分片。
- `cleanupService`：负责清理孤儿分片。
- `cleanupChunksJob`：负责定时触发清理任务。

## 9. 建议开发顺序

1. 设计后端接口。
2. 实现后端分片接收与暂存。
3. 实现后端已上传分片查询。
4. 实现后端分片合并。
5. 实现前端文件选择与拖拽。
6. 实现前端文件切片。
7. 实现前端分片并发上传。
8. 实现真实上传进度统计。
9. 实现暂停、继续、取消。
10. 实现继续上传时的缺失分片补传。
11. 实现文件 Hash 计算与秒传检查。
12. 实现后端孤儿分片清理。
13. 执行 MVP 场景验收。

## 10. MVP 验收场景

- 选择一个大文件后，可以创建上传任务。
- 拖拽一个大文件后，可以创建上传任务。
- 上传过程中进度条能真实变化。
- 分片上传时最多只允许指定数量的请求并发执行。
- 点击暂停后，正在上传的请求被中止，任务状态变为暂停。
- 点击继续后，前端先查询后端已上传分片。
- 继续上传时，只上传缺失分片。
- 上传同一个已存在文件时，后端返回已存在，前端提示秒传成功。
- 所有分片上传完成后，后端可以按序合并出完整文件。
- 上传一半后放弃，后端定时任务可以清理未合并的临时分片。
